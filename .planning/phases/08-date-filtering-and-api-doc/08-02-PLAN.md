Create `docs/API-REQUIREMENTS-V2.md` with the following structure and content. Follow the style of existing docs in the `docs/` folder (plain Markdown with endpoint specs, request/response examples, and field descriptions).

**Document structure:**

1. **Header** -- Title: "API Requirements: Usage Dashboard v2.1+" with a brief intro explaining this is a requirements handoff for the backend team, NOT implementation. State that the frontend currently uses mock data for click/QR split and this document specifies what the API needs to return for real data.

2. **Current State** -- Document the current working API:
   - Endpoint: `GET /user-activity-summary/{uid}?start_date=YYYY-MM-DD&end_date=YYYY-MM-DD`
   - Auth: `x-api-key` header
   - Current response shape: `{ source: [{ date: "YYYY-MM-DD", totalHits: int, hitCost: float, balance: float }] }`
   - Note: Also mention the `by-source` and `by-link` endpoints exist but their response shapes are not fully documented

3. **Requirement 1: Real Clicks vs QR Scans Breakdown** -- Specify:
   - Add `clicks` (int) and `qrScans` (int) fields to each daily record in the response
   - Constraint: `clicks + qrScans === totalHits` for each day
   - The frontend currently uses a 70/30 mock split (splitHits function) and will switch to real data when available
   - Suggested approach: Explore the existing `/user-activity-summary/{uid}/by-source` endpoint -- it may already contain source-level breakdown data that can be aggregated into clicks vs QR scans
   - Proposed response shape:
     ```json
     {
       "source": [{
         "date": "2026-02-20",
         "totalHits": 150,
         "clicks": 105,
         "qrScans": 45,
         "hitCost": 0.15,
         "balance": 9.85
       }]
     }
     ```

4. **Requirement 2: Other Services Data** -- Specify:
   - Currently not returned by any endpoint -- the frontend has deferred "Other Services" display for v2.0
   - When ready, the frontend needs: `{ date: "YYYY-MM-DD", description: string, amount: float }` per charge
   - Could be a new endpoint or a separate key in the existing response
   - Examples: domain renewals, wallet top-ups, premium features
   - Priority: LOW -- deferred past v2.0

5. **Requirement 3: Wallet Transaction History** -- Specify:
   - Current API returns daily `balance` (running balance per day) but no transaction events
   - For a future wallet page or expanded dashboard, need individual transaction records
   - Proposed shape: `{ date: "YYYY-MM-DD", type: "credit"|"debit", amount: float, description: string, balance_after: float }`
   - Could be a new endpoint: `GET /wallet-transactions/{uid}?start_date=X&end_date=Y`
   - Priority: LOW -- deferred past v2.0

6. **Frontend Integration Notes** -- Specify:
   - The WordPress plugin proxies all API calls through `admin-ajax.php?action=tp_get_usage_summary`
   - Server-side code: `includes/TrafficPortal/TrafficPortalApiClient.php` (getUserActivitySummary method)
   - AJAX handler: `includes/class-tp-api-handler.php` (tp_get_usage_summary handler)
   - Response reshaping: The AJAX handler validates and reshapes the API response before sending to the browser (unwraps `source` key to `days`, type-checks fields)
   - When new fields are added to the API, the AJAX handler's validation/reshaping code must be updated to pass them through
   - The JS rendering code in `assets/js/usage-dashboard.js` will need to read the new fields instead of calling `splitHits()`

7. **Migration Path** -- Brief section explaining:
   - Phase 1 (current v2.0): Frontend uses mock split, API unchanged
   - Phase 2 (v2.1): API adds `clicks`/`qrScans` to daily records, frontend removes splitHits() mock
   - Phase 3 (v2.2+): Other services and wallet transactions added

**Tone:** Technical but concise. Aimed at a backend developer who knows the API codebase. Use code blocks for all response shapes. Keep it under 200 lines.
