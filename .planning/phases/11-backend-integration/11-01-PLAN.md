<task type="auto">
  <name>Task 1: Wire wallet fetch and merge into ajax_get_usage_summary()</name>
  <files>includes/class-tp-api-handler.php</files>
  <action>
Modify the `ajax_get_usage_summary()` method in `class-tp-api-handler.php` to add wallet integration between the validation step (line ~1620) and the wp_send_json_success() call (line ~1625).

The modification inserts a nested try/catch block INSIDE the existing try block, after `$validated = $this->validate_usage_summary_response($raw)`:

1. Extract `$days = $validated['days']`
2. Open inner try block:
   - Instantiate `$walletClient = new \TerrWallet\TerrWalletClient()`
   - Get WordPress user ID via `get_current_user_id()` (NOT the Traffic Portal `$uid` -- these are different IDs)
   - Call `$transactions = $walletClient->getTransactions($wpUserId, $start_date, $end_date)`
   - Call `$days = \TerrWallet\UsageMergeAdapter::merge($days, $transactions)`
   - Add log line: `'Wallet data merged: ' . count($transactions) . ' transactions'`
3. Catch block catches ONLY `\TerrWallet\Exception\TerrWalletException $e` (not \Exception):
   - Log via `error_log('TP Link Shortener: Wallet data unavailable: ' . $e->getMessage())`
   - Also log via `$this->log_to_file('Wallet data unavailable: ' . $e->getMessage())`
   - Map null otherServices onto every day: `$days = array_map(function($day) { $day['otherServices'] = null; return $day; }, $days)`
4. After the inner try/catch, update the log line to use `count($days)` and send: `wp_send_json_success(['days' => $days])`

IMPORTANT constraints per user decisions:
- otherServices fields MUST be present (set to null) on failure, not absent -- frontend checks null, not field existence
- Catch only TerrWalletException, NOT \Exception -- merge adapter bugs must bubble up
- Use get_current_user_id() for wallet, NOT the $uid variable (which is Traffic Portal UID)
- No caching -- fresh computation per request
- No custom timeout wrapper needed -- both wallet paths are synchronous in-process PHP calls
