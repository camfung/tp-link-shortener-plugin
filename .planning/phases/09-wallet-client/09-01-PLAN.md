    1. Create `includes/TerrWallet/TerrWalletClient.php`:
       - Namespace: `TerrWallet`
       - Use statements for all three exceptions and the WalletTransaction DTO

       **Constructor:** No constructor args needed. Plugin detection and credential checks happen at call time (not construction time -- per Research pitfall 4 about plugin detection timing).

       **Public method `getTransactions(int $userId, string $afterDate, string $beforeDate): array`**
       Returns `WalletTransaction[]`. This is the single entry point.

       Logic:
       - If `function_exists('get_wallet_transactions')` is true, call `$this->fetchViaDirect($userId, $afterDate, $beforeDate)`
       - Otherwise, call `$this->fetchViaRest($userId, $afterDate, $beforeDate)`

       **Private method `fetchViaDirect(int $userId, string $afterDate, string $beforeDate): array`**
       - Call `get_wallet_transactions()` with these args:
         ```php
         [
             'user_id'  => $userId,
             'where'    => [['key' => 'type', 'value' => 'credit', 'operator' => '=']],
             'after'    => $afterDate . ' 00:00:00',
             'before'   => $beforeDate . ' 23:59:59',
             'order_by' => 'date',
             'order'    => 'ASC',
             'limit'    => '',  // No limit = all results (WCLI-03: pagination handled by function)
         ]
         ```
       - The `limit => ''` means get_wallet_transactions() returns ALL matching rows -- no manual pagination needed for the direct path
       - Map each result through `WalletTransaction::fromRaw()` and return the array
       - Filter only `type === 'credit'` via the `where` param (per project decision: credit transactions only, debits would double-count costs)

       **Private method `fetchViaRest(int $userId, string $afterDate, string $beforeDate): array`**
       This is the cron/CLI fallback per user's locked decision. Uses `rest_do_request()` (NOT wp_remote_get).

       - Check that `TP_WC_CONSUMER_KEY` and `TP_WC_CONSUMER_SECRET` constants are defined. If not, throw `TerrWalletException` with message about adding them to wp-config.php.
       - Resolve user email: `get_userdata($userId)->user_email` (per Research pitfall 2: v3 API requires email, not user ID)
       - Paginate through all results (WCLI-03):
         ```php
         $allTransactions = [];
         $page = 1;
         $perPage = 100;
         do {
             $request = new \WP_REST_Request('GET', '/wc/v3/wallet');
             $request->set_query_params([
                 'email'           => $userEmail,
                 'per_page'        => $perPage,
                 'page'            => $page,
                 'consumer_key'    => TP_WC_CONSUMER_KEY,
                 'consumer_secret' => TP_WC_CONSUMER_SECRET,
             ]);
             $response = rest_do_request($request);

             if ($response->is_error()) {
                 $error = $response->as_error();
                 throw new TerrWalletApiException(
                     'TerrWallet REST API error: ' . $error->get_error_message()
                 );
             }

             $data = $response->get_data();
             $allTransactions = array_merge($allTransactions, $data);
             $page++;
         } while (count($data) === $perPage);
         ```
       - Filter to credit transactions only: `array_filter` where `$item['type'] === 'credit'`
       - Filter by date range in PHP (per Research pitfall 6: REST API has no native date filtering):
         - Extract date portion (Y-m-d) from each transaction's `date` field
         - Keep only transactions where date >= $afterDate AND date <= $beforeDate
       - Map each through `WalletTransaction::fromRaw()` (cast to object if array)
       - Return the array
